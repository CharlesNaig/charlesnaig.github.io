How it works

From tap to recorded

A student taps their NFC card. In under 3.5 seconds, TamTap reads, verifies, photographs, detects, records, and broadcasts — all on-premise.

01

NFC Tap

RC522 reader via SPI reads the student's MIFARE card UID in under 100ms.

02

Validate

Lookup student in MongoDB (or JSON fallback). Check schedule and grace period.

03

Capture

Pi Camera v2 fires via rpicam-still subprocess. Photo saved by date.

04

Face Detect

OpenCV Haar cascade checks for a face ≥ 80×80px. Detection only — no recognition.

05

Record

Attendance saved to MongoDB. Green LED + buzzer confirm success.

06

Broadcast

Socket.IO pushes attendance:new event to all connected dashboard clients on LAN.

System design

Architecture

Four distinct layers communicate cleanly — hardware speaks to Python, Python speaks to Node.js, Node.js speaks to the browser and MongoDB.

⬡ Hardware Layer

Raspberry Pi 4B

RC522 NFC reader over SPI, Pi Camera v2 over CSI, I2C LCD 16×2 display, GPIO LEDs, buzzer, and three physical control buttons.

SPII2CCSI GPIO 17/27/18Bookworm OS
◈ Python Layer

tamtap.py + database.py

State machine running IDLE → CARD_DETECTED → CAMERA_ACTIVE → SUCCESS/FAIL. Calls database.py for unified MongoDB/JSON access. Posts to server via HTTP.

Python 3.11State machine mfrc522OpenCV
◇ Backend Layer

server.js (Express + Socket.IO)

REST API with 10 route modules, session-based auth with bcrypt, hardware API key guard, real-time Socket.IO broadcasting, and live systemd log streaming.

Node.js 20Express Socket.IOconnect-mongo
○ Frontend Layer

Multi-page HTML Dashboard

No frameworks. Vanilla ES6+ with Fetch API, Tailwind CSS via CDN, Chart.js for analytics, SweetAlert2 for modals, and Socket.IO client for live updates.

Vanilla JSTailwind CSS Chart.jsFetch API
⬟ Data Layer

MongoDB (Local) + JSON Fallback

Seven collections — students, teachers, admins, attendance, schedules, calendar, settings — all running locally on the Pi. When MongoDB is unreachable, the Python layer falls back to tamtap_users.json transparently.

MongoDB 7+Local instance 7 collectionsJSON fallback Offline mode
Components

Hardware specs

All processing happens on the Pi. No external compute, no cloud.

Raspberry Pi 4B

4GB RAM · Bookworm OS

Main compute unit hosting the NFC loop, Node.js server, and MongoDB — all simultaneously on a single board.

RC522 NFC Reader

SPI · GPIO 8,9,10,11,25

Reads MIFARE card UIDs. Connected via SPI bus. Poll interval 100ms for responsive tap detection.

Pi Camera v2

5MP · CSI interface

Snapshot-only via rpicam-still subprocess. Photos stored by date for attendance records.

I2C LCD 16×2

I2C · Address 0x27

Displays tap feedback in real time — student name, status (PRESENT/LATE/FAIL), and timestamp.

LEDs + Buzzer

GPIO 17 (green) · 27 (red) · 18 (buzzer)

Green LED + buzzer tone on success. Red LED + different tone on failure. Via relay module.

Control Buttons

GPIO 5 (start) · 6 (restart) · 13 (stop)

Physical buttons with active LOW pull-ups. Stop requires a 2.5s hold to prevent accidental shutdown.

Core logic

State machine

Every attendance cycle follows a strict linear state flow. No skipping, no parallel transitions — every cycle returns to IDLE within 3.5 seconds.

IDLE
Polling NFC every 100ms
→
CARD_DETECTED
UID read, DB lookup
→
CAMERA_ACTIVE
Photo capture ≤1.5s
↗
SUCCESS
Green LED + POST
↘
FAIL
Red LED + buzzer
# Timing budget breakdown (total ≤ 3.5 seconds) NFC_POLL_INTERVAL = 0.1 # s — 100ms NFC read loop SCHEDULE_VALIDATE = ~0.05 # s — DB lookup + schedule check CAMERA_CAPTURE_TIME = 1200 # ms — rpicam-still shutter CAMERA_TIMEOUT = 2.5 # s — subprocess hard cap FACE_DETECTION_TIMEOUT = 1.0 # s — Haar cascade cap MIN_FACE_SIZE = (80, 80) # px — minimum face dimensions LED_LCD_UPDATE = ~0.1 # s — feedback output
